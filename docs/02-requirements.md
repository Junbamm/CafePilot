# 02. 요구사항 정의 (Requirements)

본 문서는 [01-project-overview.md](./01-project-overview.md)에서 정의한 MVP 범위를 기능 단위로 구체화한다.

우선순위는 MoSCoW 기법을 따른다.

- **Must**: MVP에 반드시 포함되어야 하는 기능
- **Should**: 가능하면 포함하되, 일정상 후순위로 미룰 수 있는 기능
- **Could**: 여유가 있을 때 추가하는 기능

## 1. 권한 정책 개요

| 역할 | 설명 |
|---|---|
| `ROLE_OWNER` | 카페를 등록/소유한 사장. 자신의 카페에 대한 모든 관리 권한을 가진다. |
| `ROLE_STAFF` | 카페 소속 직원. 주문 처리, 재고 확인 등 제한된 운영 권한을 가진다. |
| `ROLE_ADMIN` | 플랫폼 시스템 관리자. 전체 사용자, 운영 로그, 통계에 접근한다. |

> 상세 권한 매트릭스(리소스별 CRUD 권한)는 `05-api-spec.md`에서 엔드포인트별로 명시한다.

## 2. 기능 요구사항 (Functional Requirements)

### 2.1 회원 (Auth / Member)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-AUTH-01 | 사용자는 이메일/비밀번호로 회원가입할 수 있다. | Must |
| FR-AUTH-02 | 회원가입 시 이메일 중복은 허용하지 않는다. | Must |
| FR-AUTH-03 | 비밀번호는 안전하게 해시되어 저장된다. | Must |
| FR-AUTH-04 | 사용자는 이메일/비밀번호로 로그인할 수 있다. | Must |
| FR-AUTH-05 | 로그인 성공 시 JWT Access Token이 발급된다. | Must |
| FR-AUTH-06 | Access Token 만료 시 Refresh Token으로 재발급받을 수 있다. | Should |
| FR-AUTH-07 | 사용자는 OWNER 또는 STAFF 역할 중 하나를 가진다. | Must |
| FR-AUTH-08 | 인증되지 않은 사용자는 보호된 API에 접근할 수 없다. | Must |
| FR-AUTH-09 | 권한이 없는 사용자는 허용되지 않은 API에 접근할 수 없다(403). | Must |

### 2.2 카페 (Cafe)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-CAFE-01 | OWNER는 카페를 등록할 수 있다. | Must |
| FR-CAFE-02 | OWNER는 자신이 등록한 카페 정보를 조회할 수 있다. | Must |
| FR-CAFE-03 | OWNER는 자신이 등록한 카페 정보를 수정할 수 있다. | Must |
| FR-CAFE-04 | 하나의 OWNER 계정은 여러 카페를 등록할 수 있다. | Should |
| FR-CAFE-05 | STAFF는 자신이 소속된 카페 정보를 조회할 수 있다. | Should |

### 2.3 메뉴 (Menu)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-MENU-01 | OWNER/STAFF는 카페에 메뉴를 등록할 수 있다. | Must |
| FR-MENU-02 | OWNER/STAFF는 메뉴 정보(이름, 가격, 설명 등)를 수정할 수 있다. | Must |
| FR-MENU-03 | OWNER/STAFF는 메뉴를 삭제(또는 판매 중지)할 수 있다. | Must |
| FR-MENU-04 | 메뉴는 카페 단위로 목록 조회할 수 있다. | Must |
| FR-MENU-05 | 메뉴는 재고와 연결되어, 재고가 0이면 품절 상태로 표시된다. | Should |

### 2.4 주문 (Order)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-ORDER-01 | STAFF/OWNER는 신규 주문을 생성할 수 있다. | Must |
| FR-ORDER-02 | 주문은 하나 이상의 메뉴 항목과 수량을 포함한다. | Must |
| FR-ORDER-03 | 주문 생성 시 해당 메뉴의 재고가 자동으로 차감된다. | Must |
| FR-ORDER-04 | 재고가 부족하면 주문 생성이 거부된다. | Must |
| FR-ORDER-05 | 주문 상태는 접수 → 처리중 → 완료(또는 취소)로 전이된다. | Must |
| FR-ORDER-06 | OWNER/STAFF는 카페 단위로 주문 목록을 조회할 수 있다. | Must |
| FR-ORDER-07 | OWNER/STAFF는 주문 상태를 변경할 수 있다. | Must |
| FR-ORDER-08 | 주문 생성/상태 변경 시 이벤트가 발행되어 통계 집계에 활용된다. | Should |

### 2.5 재고 (Inventory)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-INV-01 | OWNER/STAFF는 재고 품목을 등록할 수 있다. | Must |
| FR-INV-02 | 재고는 주문 발생 시 자동으로 차감된다. | Must |
| FR-INV-03 | 재고 수량이 임계치 이하로 떨어지면 부족 알림 이벤트가 발행된다. | Must |
| FR-INV-04 | OWNER/STAFF는 재고 현황을 조회할 수 있다. | Must |
| FR-INV-05 | OWNER/STAFF는 재고를 수동으로 보충(입고)할 수 있다. | Should |

### 2.6 매출 (Sales & Statistics)

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-SALES-01 | OWNER는 일별 매출 통계를 조회할 수 있다. | Must |
| FR-SALES-02 | OWNER는 월별 매출 통계를 조회할 수 있다. | Must |
| FR-SALES-03 | OWNER는 기간별 인기 메뉴 순위를 조회할 수 있다. | Must |
| FR-SALES-04 | 매출 통계는 완료된 주문 데이터를 기준으로 집계된다. | Must |
| FR-SALES-05 | 통계 조회 성능을 위해 캐싱이 적용된다. | Should |

### 2.7 AI 질의응답

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-AI-01 | 사용자는 자연어로 운영 데이터에 대해 질의할 수 있다. | Must |
| FR-AI-02 | AI는 실제 매출/주문/재고 DB 데이터를 근거로 답변한다. | Must |
| FR-AI-03 | AI는 오늘 매출 요약을 제공할 수 있다. | Must |
| FR-AI-04 | AI는 가장 많이 팔린 메뉴를 제공할 수 있다. | Must |
| FR-AI-05 | AI는 재고 부족 품목을 제공할 수 있다. | Must |
| FR-AI-06 | AI는 데이터 기반 발주 추천을 제공할 수 있다. | Should |
| FR-AI-07 | AI는 매출 추세 분석을 제공할 수 있다. | Should |
| FR-AI-08 | AI는 운영 개선 제안을 제공할 수 있다. | Could |
| FR-AI-09 | AI 응답 근거가 된 데이터 범위(기간 등)가 함께 표시된다. | Should |

|---|---|---|

| FR-EVT-02 | 재고 부족 알림은 이벤트 구독자(알림 처리기)를 통해 처리된다. | Must |
| FR-EVT-03 | 매출 집계는 스케줄러를 통해 주기적으로 자동 수행된다. | Should |
| FR-EVT-04 | 이벤트 처리 실패 시 재시도 또는 실패 로그가 기록된다. | Should |

| ID | 요구사항 | 우선순위 |
|---|---|---|
| FR-ADMIN-01 | ADMIN은 전체 사용자 목록을 조회할 수 있다. | Must |
| FR-ADMIN-02 | ADMIN은 사용자 계정을 활성화/비활성화할 수 있다. | Should |
| FR-ADMIN-03 | ADMIN은 시스템 운영 로그를 조회할 수 있다. | Must |

## 3. 비기능 요구사항 (Non-Functional Requirements)

| 구분 | 요구사항 |
|---|---|
| 보안 | 비밀번호는 단방향 해시(BCrypt)로 저장하며, 평문으로 저장/로깅하지 않는다. |
| 보안 | 모든 인증이 필요한 API는 JWT 기반으로 검증되며, 역할(Role) 기반 접근 제어를 적용한다. |
| 데이터 무결성 | 재고 차감은 동시 주문 상황에서도 정합성이 깨지지 않아야 한다(동시성 제어 필요). |
| 성능 | 자주 조회되는 매출 통계는 Redis 캐싱을 통해 응답 속도를 개선한다. |
| 가용성 | 이벤트 처리(알림, 통계 집계)는 비동기로 처리되어 API 응답 지연을 유발하지 않는다. |
| 확장성 | 도메인 간 결합도를 낮춰 신규 기능(예: 결제 연동) 추가 시 기존 코드 변경을 최소화한다. |
| 운영성 | 주요 비즈니스 예외와 시스템 오류는 공통 응답 포맷으로 일관되게 반환된다. |
| 운영성 | 모든 API는 Swagger(OpenAPI)로 문서화되어 별도 설명 없이 테스트 가능해야 한다. |
| 테스트 | 핵심 비즈니스 로직(주문 생성, 재고 차감, 매출 집계)은 단위 테스트로 검증한다. |
| 배포 | 애플리케이션과 인프라(MySQL, Redis, RabbitMQ)는 Docker Compose로 한 번에 기동 가능해야 한다. |

## 4. 다음 단계와의 연결

- 위 기능 요구사항의 명사(카페, 메뉴, 주문, 재고, 매출, 회원)는 `03-domain.md`에서 도메인 모델로 구체화된다.
- 각 FR은 `05-api-spec.md`의 엔드포인트와 1:1 또는 1:N으로 매핑된다.
- 비기능 요구사항 중 아키텍처에 영향을 주는 항목(캐싱, 비동기 처리, 동시성 제어)은 `06-architecture.md`에서 구체적인 해결 방식을 다룬다.
