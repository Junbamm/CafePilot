# 01. 프로젝트 개요 (Project Overview)

## 1. 프로젝트 소개

**CafePilot**은 카페 사장을 위한 AI 기반 운영 지원 SaaS 플랫폼이다.

단순한 AI 챗봇이 아니라, 카페 운영에 필요한 매출·주문·재고 데이터를 하나의 시스템에서 통합 관리하고, 그 데이터를 기반으로 AI가 운영 의사결정을 지원하는 **백엔드 중심 서비스**다.

## 2. 배경 및 문제 정의

소규모 카페 운영자는 다음과 같은 문제를 공통적으로 겪는다.

| 문제 | 설명 |
|---|---|
| 시스템 파편화 | POS, 재고 관리, 매출 정산을 서로 다른 시스템/엑셀로 관리한다. |
| 데이터 분산 | 매출·주문 데이터가 여러 곳에 흩어져 있어 통합 조회가 어렵다. |
| 수기 재고 관리 | 재고를 수기로 기록하다 보니 부족/누락이 자주 발생한다. |
| 분석 부재 | 데이터는 쌓이지만 이를 분석해 운영에 반영하기 어렵다. |
| 의사결정 지원 부재 | "무엇을 더 팔아야 하는지", "언제 발주해야 하는지"에 대한 근거 있는 답을 얻기 어렵다. |

CafePilot은 위 문제를 **하나의 통합 백엔드 플랫폼 + AI 데이터 분석**으로 해결하는 것을 목표로 한다.

## 3. 프로젝트 목표

### 3.1 서비스 목표

- 카페 운영에 필요한 핵심 데이터(카페, 메뉴, 주문, 재고, 매출)를 하나의 시스템에서 관리한다.
- 운영 데이터를 기반으로 AI가 의미 있는 인사이트(매출 요약, 인기 메뉴, 재고 부족, 발주 추천 등)를 제공한다.
- 이벤트 기반 비동기 처리와 캐싱, 스케줄러를 통해 실 서비스 수준의 운영 안정성을 갖춘다.

### 3.2 엔지니어링 목표

이 프로젝트는 이직 포트폴리오를 겸하므로, 기능 구현 자체보다 아래 역량을 증명하는 것을 목표로 한다.

- **설계 능력**: 요구사항 정의 → 도메인 설계 → ERD → API 설계 → 아키텍처 순서로 진행하는 체계적인 설계 프로세스
- **객체지향 설계**: 계층 간 책임 분리, DTO/Entity 분리, SOLID 원칙 적용
- **코드 품질**: 생성자 주입, Validation, 계층형 아키텍처, 공통 응답/예외 처리
- **테스트**: 단위 테스트와 통합 테스트를 통한 품질 검증
- **운영 경험**: Docker 기반 배포, CI/CD, 로깅, 장애 대응(트러블슈팅 문서화)
- **아키텍처 이해도**: 이벤트 기반 비동기 처리(RabbitMQ), 캐싱(Redis), 스케줄링 등 실무형 아키텍처 구성 요소 활용

## 4. 핵심 가치 제안 (Value Proposition)

> "여러 시스템에 흩어진 카페 운영 데이터를 하나로 모으고, AI가 그 데이터를 해석해서 다음 행동을 제안한다."

- 사장은 여러 툴을 오갈 필요 없이 CafePilot 하나로 주문·재고·매출을 관리한다.
- AI는 단순 챗봇이 아니라, 실제 운영 데이터(DB)를 근거로 답변하는 **데이터 기반 의사결정 지원 도구**다.

## 5. 대상 사용자

| 사용자 유형 | 설명 |
|---|---|
| 카페 사장(Owner) | 카페를 등록하고 메뉴/재고/매출을 관리하며 AI 인사이트를 활용하는 주 사용자 |
| 매니저/직원(Staff) | 제한된 권한으로 주문 처리, 재고 확인 등 운영 업무를 수행하는 사용자 |
| 시스템 관리자(Admin) | 플랫폼 전체 사용자와 운영 로그, 통계를 관리하는 운영자 |

## 6. MVP 범위

### 6.1 In Scope (MVP 포함)

- 회원가입 / 로그인 / JWT 인증 / 권한 관리
- 카페 등록 / 조회 / 수정
- 메뉴 등록 / 수정 / 삭제 / 조회
- 주문 생성 / 조회 / 상태 변경
- 재고 등록 / 차감 / 부족 알림
- 일별·월별 매출 통계, 인기 메뉴 분석
- 운영 데이터 기반 AI 질의응답 (매출 요약, 재고 부족, 발주 추천, 추세 분석, 개선 제안)
- RabbitMQ 기반 비동기 이벤트 처리
- Redis 캐싱
- Scheduler 기반 자동 작업(예: 정기 재고 점검, 일일 매출 집계)
- 관리자 기능(사용자 관리, 운영 로그 조회, 통계 조회)

### 6.2 Out of Scope (MVP 제외, 추후 검토)

- 실시간 결제 연동(PG사 연동)
- 다국어/다통화 지원
- 프랜차이즈(다중 지점) 통합 대시보드
- 모바일 네이티브 앱
- 고객(소비자)向 주문 앱 — CafePilot은 사장/직원을 위한 운영 지원 시스템이며, 소비자向 주문 채널은 범위 밖이다.
- 자체 LLM 파인튜닝 — 외부 LLM API를 데이터 기반 프롬프트로 활용하는 방식을 채택하고, 모델 자체 학습은 다루지 않는다.

## 8. 프로젝트 성격

CafePilot은 두 가지 목표를 동시에 가진다.

1. **실 서비스 가능한 MVP**: 실제 카페 사장이 사용해도 무리 없는 수준의 기능적 완성도와 운영 안정성을 갖춘다.
2. **이직 포트폴리오**: 단순히 동작하는 것을 넘어, 설계 문서화·아키텍처 결정·테스트·운영 고려사항을 갖춰 면접관이 지원자의 실무 역량을 확인할 수 있는 근거 자료가 된다.

따라서 모든 설계 및 구현 결정은 "왜 이렇게 했는가"에 대한 답을 문서(`docs/`)에 남기는 것을 원칙으로 한다.

## 9. 성공 기준

- [ ] 핵심 도메인(회원/카페/메뉴/주문/재고/매출)이 명확한 책임 경계를 가지고 설계되어 있다.
- [ ] 인증/인가가 JWT 기반으로 안전하게 동작한다.
- [ ] AI 기능이 실제 DB 데이터를 근거로 응답한다.
- [ ] 이벤트 기반 처리(RabbitMQ), 캐싱(Redis), 스케줄러가 실제로 동작한다.
- [ ] 단위/통합 테스트가 핵심 비즈니스 로직을 검증한다.
- [ ] Docker Compose로 전체 시스템(App, PostgreSQL, Redis, RabbitMQ)이 한 번에 기동된다.
- [ ] CI/CD 파이프라인이 구성되어 있다.
- [ ] 설계 문서(`docs/`)가 요구사항 → 도메인 → ERD → API → 아키텍처 순으로 일관되게 정리되어 있다.
